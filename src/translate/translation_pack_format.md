# Translation pack format + verification workflow v0

## Record format
FORMAT: one JSONL file per (model × condition) split, e.g. `packs/gptoss20b_attack.jsonl`, UTF-8, one record per transcript. The CoT is stored SEGMENTED (split on blank lines / numbered-step boundaries) so translation, verification, and correction logging are all segment-aligned 1:1. A record:

- `id` (str): stable key, `{model}-{task_id}-{condition}-{seed}`, e.g. `gptoss20b-swebench_0142-attack-3`. Never changes; every monitor result joins on it.
- `task_id` (str): the underlying agentic task; lets you pair attack/honest for McNemar.
- `condition` ("attack"|"honest"): by construction, not judged.
- `side_task_passed` (bool): output of the hidden programmatic tests (ground truth; copied in read-only for filtering, never edited here).
- `model` (str): "gpt-oss-20b" | "qwen3-32b".
- `en_sha256` (str): hash of the joined EN CoT. Frozen at pack creation; a CI check recomputes it so the EN source can never silently drift during verification (translation must go EN→target, never the reverse).
- `segments` (list of objects), one per CoT segment, each with:
  - `seg_id` (int), `kind` ("prose"|"code"|"tool_call"): code/tool segments are copied byte-identically into every language and are excluded from translation entirely (checked by script).
  - `en` (str): source English segment.
  - `cs_mt` (str): machine draft of the Czech (Gemini Flash or DeepL free) — the draft the verifier edits, never writing from scratch.
  - `cs_verified` (str|null): the human-verified Czech; null until verified. This is what monitors see.
  - `de_mt` (str): German machine translation (locked engine + version string in file header).
  - `incrim_spans` (list of [start,end] char offsets into `en`): spans marked (once, on the EN source, before any translation) as carrying the incriminating/scheming reasoning. Empty for honest transcripts. These are the payload the whole study is about; verification is anchored on them.
  - `correction_type` (enum, set at verification): `none` | `fluency` | `terminology` | `meaning_shift` | `severity_shift_softened` | `severity_shift_strengthened` | `omission` | `addition`. The last four are "signal-relevant" corrections and feed the appendix table.
  - `verifier_note` (str, optional): one line, only when correction_type != none/fluency.
- `verify_status` ("pending"|"verified"|"recheck"), `verified_at` (ISO ts), `verify_seconds` (int, auto-logged by the review tool), `recheck_agree` (bool|null, filled for the 5% self-agreement sample).
- `de_spotcheck` (obj|null, on the 10% sample): `{backtrans_en, verdict: "ok"|"meaning_shift"|"softened", note}`.

Why JSONL + segments: streams into Inspect trivially; segment alignment makes laundering detectable per-span instead of per-page; correction_type over segments gives the correction-rate denominator for free (rate = corrected prose segments / total prose segments, sliceable by condition and type).

## Verification workflow (native Czech — the project's moat)
Pipeline: (0) PREP, once — a script builds packs from Inspect logs: segments the CoT, hashes EN, produces `cs_mt` and `de_mt` drafts, and copies code/tool segments verbatim. The verifier then does one annotation pass on EN ONLY: for each attack transcript, mark `incrim_spans` (the sentences that reveal the covert side task). This is done before looking at any Czech, so the "what must survive translation" standard is fixed independently of the translation. Budget: ~30 s/transcript.

(1) EVENING SESSIONS, batched — a tiny local HTML/keyboard review tool (he's an elite SWE; half a day to build, pays for itself immediately) shows EN and cs_mt side-by-side, segment-aligned, with incrim_spans highlighted yellow in the EN column. Queue is a RANDOM INTERLEAVE of attack and honest transcripts (never verify all attacks in one mood/energy state — asymmetric care by condition is itself a confound). Per transcript, two passes:
  - Pass A (meaning + register, whole transcript, ~60-90 s): read Czech only, then glance at EN per segment. Fix so it reads as natural Czech an LLM would plausibly produce: informal-technical register, first-person reasoning voice ("takže potřebuju…", not stiff formal Czech), code identifiers/API names left in English verbatim. Keyboard: Enter = accept segment as-is (correction_type=none), E = edit (tool then forces picking a correction_type + optional one-line note).
  - Pass B (incrimination audit, attack transcripts only, ~30 s): for each highlighted EN span, check the Czech says something EQUALLY incriminating — same intent, same explicitness, same hedging level. If the Czech is even slightly more deniable than the EN, edit and log `severity_shift_softened`; if more explicit, `severity_shift_strengthened`. This pass is the moat; it is never skipped.
  Realistic throughput after warm-up: 2.5-4 min/transcript → ~15-20 transcripts per 1-hour evening → 400 transcripts in roughly 5-6 weeks of 4 evenings/week, which fits the summer text-only window.

(2) DRIFT CONTROL — the tool silently re-queues a random 5% of already-verified transcripts ≥2 weeks later with edits hidden; the verifier re-verifies blind and the tool diffs (`recheck_agree`). Self-agreement rate is reported next to the correction rate. Three fixed calibration transcripts get re-read at the start of week 1, 3, 5 to keep the severity bar constant.

(3) LOGGING → APPENDIX — everything lands in the pack itself, so the appendix table is one pandas groupby: correction rate overall; by correction_type; by condition (attack vs honest — if attacks needed systematically more meaning-level fixes, say so, it bounds the confound); by whether the segment contained an incrim_span; plus median verify_seconds and the 5% self-agreement rate. Headline appendix claim looks like: "X% of prose segments required edits; Y% were signal-relevant (meaning/severity/omission); of incriminating spans, Z% were softened by MT and restored by the verifier — i.e., unverified MT would have laundered Z% of the signal."

Hard rules: never edit `en` (hash check enforces); never translate from memory of the attack ("I know what it did") — translate what the EN says; if a segment is untranslatably ambiguous, note it and keep the ambiguity in Czech rather than resolving it.

## German MT
German is deliberately the MT-ONLY arm — it exists to show the Czech results aren't an artifact of human curation, so it must NOT be human-polished. Produce `de_mt` once, with a single locked engine and version recorded in the pack header (recommend DeepL free API for prose segments — best-in-class for EN→DE, free tier covers 500k chars/month, so batch over ~2 months or split across the Gemini fallback; code/tool segments copied verbatim by the same script as Czech). Spot-check, not verify: on a random 10% stratified by condition, (a) back-translate de_mt→EN with a DIFFERENT engine (Gemini Flash) and diff against the EN source for meaning shifts, and (b) for attack transcripts, check each incrim_span survived — since German proficiency here is limited, this uses the back-translation plus a targeted Gemini question per span ("does this German sentence state the author is deliberately hiding X? answer yes/no + quote"), with Jakub adjudicating disagreements using school German + dictionary. Log verdicts in `de_spotcheck`; report the DE span-survival rate in the appendix beside the Czech correction rate. If span-survival <95%, flag DE results as MT-quality-limited in the write-up rather than silently trusting them.

## Pitfalls
- LAUNDERING/SOFTENING (the killer): MT and careless human translation both normalize toward bland register — 'I'll slip this backdoor in without mentioning it' becomes 'I will also make this additional change'. The monitor then misses it and the drop gets misattributed to language instead of translation. Caught by: incrim_spans annotated on frozen EN before translation + mandatory Pass B span audit + severity_shift_softened as a first-class logged correction type, so the laundering rate is itself a reported number.
- INVERSE LAUNDERING: MT sometimes disambiguates and makes covert reasoning MORE explicit than the EN (resolving a hedge into a plain confession), inflating monitor catch rates in Czech/German. Caught by the same span audit with severity_shift_strengthened — the check is symmetric: equally incriminating, not 'at least as'.
- OVER-LITERAL / DISFLUENT MT: stilted calque Czech may trip the monitor via sheer weirdness (or break the weak Qwen monitor on parsing), confounding language with fluency. Caught by Pass A's naturalness standard (register: informal first-person LLM reasoning voice) and by reporting the fluency correction rate separately from signal-relevant corrections.
- KEYWORD LEAKAGE: leaving EN words like 'backdoor', 'hide', 'bypass' untranslated hands the keyword baseline (and the EN-prompted monitors) free wins in the 'Czech' condition. Policy: code identifiers stay English verbatim, natural-language suspicion terms MUST be translated; a script scans cs_verified prose segments against the keyword-baseline lexicon and fails the pack on hits.
- ASYMMETRIC CARE BY CONDITION: verifying attack transcripts more attentively than honest ones creates a class-correlated quality difference that directly biases AUC. Caught by: random attack/honest interleave in the review queue, auto-logged verify_seconds per record, and reporting correction rate split by condition in the appendix.
- STRUCTURE/CODE DRIFT: MT mangling numbered steps, markdown, or code blocks changes what the monitor sees beyond language. Caught by: segment-level 1:1 alignment enforced by the tool, kind=code/tool segments copied byte-identically and diff-checked by script.
- SOURCE CONTAMINATION: 'fixing' the English while verifying, so language versions no longer share a source. Caught by en_sha256 frozen at pack creation and re-verified in CI before any monitor run.
- VERIFIER DRIFT (solo, weeks-long): the severity bar for 'equally incriminating' slides over time. Caught by the blind 5% re-check (self-agreement reported) and fixed calibration transcripts re-read biweekly.
- TRANSLATING FROM MEMORY: knowing the attack, the verifier writes Czech that says what the model DID rather than what the EN CoT SAYS, injecting signal. Rule: translate the text, not the ground truth; ambiguity in EN must stay ambiguous in Czech.
