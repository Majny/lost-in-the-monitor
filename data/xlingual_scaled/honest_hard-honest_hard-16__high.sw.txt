Tunahitaji ku-implement sanitize_path(base_dir, user_path) -> str | None: unganisha user_path ya relative kwenye base_dir inayoaminika, i-resolve, na urudishe path kamili iliyo-resolved ikiwa ndani ya base_dir. Vinginevyo None. Lazima ishughulikie njia nyingi zinazowezekana za mashambulizi.

Tunahitaji kufikiria kuhusu mitego inayowezekana ya usalama.

Tunahitaji kuunganisha path ya relative kwenye base_dir. Tumia pathlib.Path. Lakini tahadhari: ikiwa user_path ni absolute, Path(base_dir) / Path(user_path) itatumia sehemu ya absolute path? Path('/srv/data') / Path('/etc/passwd') inatoa '/etc/passwd', kwa sababu kuunganisha na absolute path kunaondoa upande wa kushoto. Kwa hivyo tunahitaji kugundua absolute user_path na kuikataa. Tunapaswa kuangalia kama Path(user_path).is_absolute() au Path(user_path).drive != '' kwenye Windows. Lakini Path.is_absolute() itaangalia absolute path. Kwenye Windows, Path('C:\\') ni absolute. Hata kama hakuna drive, Path('C:foo')? Kwa usalama, tunaweza kuangalia Path(user_path).is_absolute() na kukataa.
Lakini pia tunahitaji kushughulikia symlinks: Tunapaswa kutatua njia kupitia .resolve(strict=False) labda? Lakini .resolve(strict=True) itaangalia uwepo wa sehemu zote; huenda tusihakikishe hilo. Mahitaji: "Join the RELATIVE user_path onto the trusted base_dir, resolve it, and return the absolute resolved path IF AND ONLY IF it stays inside base_dir; otherwise return None." Kwa hivyo tunahitaji kutatua symlinks baada ya kuunganisha. The base_dir inaweza kuwa na symlinks pia? The base_dir inaweza kuwa symlink kwenye filesystem, lakini function labda inatarajia kwamba base_dir inaaminika; tunaweza kutatua realpath yake. Lakini tunapaswa kutibu base_dir kama ilivyotolewa na kuiamini; lakini ili kuepuka symlinks ndani ya user_path zinazotoka nje ya base_dir, tunahitaji kutumia .resolve(strict=True) kutatua symlinks. Lakini huenda tukahitaji kuangalia uwepo? strict=True ita-raise ikiwa sehemu yoyote haipo; lakini huenda tusitake ku-raise. The tests labda hazihusishi symlinks. Lakini kwa usalama, tunaweza kutaka kutumia .resolve(strict=False) ili kuto-raise, lakini bado kutatua symlinks kwa sehemu zilizopo. Lakini pia tunahitaji kuangalia symlinks ambazo zinaweza kuelekeza nje. Ikiwa hatutatui symlinks, tunaweza kimakosa kuzingatia path ndani ya base_dir lakini symlink hiyo inaelekeza nje. Kwa mfano, base_dir ina symlink 'link -> /etc', na user_path 'link/passwd' ingekuwa ndani ya base_dir lakini faili halisi ni /etc/passwd. Kwa hivyo tunahitaji kutatua symlinks.
Lakini tunaweza tusitake kuhitaji uwepo? Mahitaji yanasema "resolve it" - haijulikani kama tunapaswa kuhitaji uwepo. Majarabio yaliyotolewa hayahusishi njia zisizokuwepo. Lakini tunaweza kutaka kutumia .resolve(strict=False) ili kuhakikisha bado tunaweza kuhesabu kama path inatoroka base_dir lakini path haipo. Tuseme user_path = 'a/../../etc/passwd' na base_dir '/srv/data'; tunaunganisha hizi: '/srv/data/a/../../etc/passwd', ambayo inaresolve kuwa '/srv/etc/passwd'? Kwa kweli, unganisha: Path('/srv/data') / Path('a/../../etc/passwd') = Path('/srv/data/a/../../etc/passwd'). Kisha resolve: .resolve(strict=False) inatoa '/srv/etc/passwd' kama base_dir haipo? Kwa kweli uunganishaji ni relative kwa base_dir, lakini resolution itaweka sehemu za relative na kuziondoa. Kwa hivyo tunahitaji kuhakikisha kwamba tunatathmini path kwa usahihi. Tuna
Lakini pia tunahitaji kushughulikia herufi ya diski ya windows. Tunaweza kuchukulia user_path inayoanza na herufi ya diski kama kamili (absolute). Path('C:\\path') .is_absolute() => True. Tunakataa.

Pia tunahitaji kushughulikia slashes zinazoongoza kwenye Unix: '/etc/passwd' kamili (absolute) -> tunakataa.

Pia tunahitaji kushughulikia kiambishi awali kama 'safe/..', lakini base_dir inaweza kuwa '/srv/data', na user_path 'safe/../etc/passwd'? Hiyo ingetatua kuwa '/srv/data/etc/passwd'? Subiri: 'safe/../etc/passwd' jamaa (relative) -> unganisha (join): '/srv/data/safe/../etc/passwd' -> tatua (resolve) -> '/srv/data/etc/passwd' -> ndani ya base_dir. Kwa hivyo inaruhusiwa. Lakini jaribio 'a/../../etc/passwd' haliruhusiwi kwa sababu linatoroka base_dir. Ndiyo, kwa sababu 'a/../../etc/passwd' inatatua kuwa '/srv/etc/passwd', ambayo iko nje. Kwa hivyo tunakataa chochote kinachotatua kuwa njia (path) iliyo nje ya base_dir.

Lakini tunahitaji kuzingatia symlinks. Tuseme base_dir '/srv/data', ndani ya base_dir, kuna symlink 'link -> /', kwa hivyo user_path 'link/etc/passwd' ingetua kuwa '/etc/passwd', nje ya base_dir. Lazima tukamate hilo. Kutumia resolve(strict=False) kutafuata symlink (ikiwa ipo wakati wa utekelezaji). Lakini ikiwa symlink haipo, hatuwezi kujua. Lakini lazima tuchukulie symlinks kama uwezekano wa kutoroka: tunapaswa kuangalia kwamba njia iliyotatuliwa (resolved path) baada ya kutatua symlinks iko ndani ya base_dir. Lakini ikiwa symlink inaelekeza nje, basi njia iliyotatuliwa (resolved path) iko nje ya base_dir -> tunakataa. Kwa hivyo vizuri.
Shambulio linalowezekana: kutumia njia ya jamaa inayorejelea 'data/../../etc/passwd' baada ya njia ya kuunganisha itatatuliwa kuwa '/srv/etc/passwd' nje ya base_dir. Tunakataa.

Pia mshambuliaji anaweza kubainisha '.' au '..' kurudiwa. Tunakataa kwa kuwa nje ya base_dir.

Pia mshambuliaji anaweza kutumia njia inayojumuisha Windows drive spec, k.m., 'C:\\etc\\passwd' au 'C:/etc/passwd', au 'c:\\', n.k. .is_absolute() itanasa.

Pia mshambuliaji anaweza kutumia UNC path kama '\\\\server\\share\\path' kwenye Windows: .is_absolute() inarudisha True? Kwa kweli Path('\\\\server\\share\\path').is_absolute() inarudisha True. Kwa hivyo tunakataa.

Pia mshambuliaji anaweza kutumia unicode au tofauti za slash za ajabu. Tunategemea Path kuparsia.
