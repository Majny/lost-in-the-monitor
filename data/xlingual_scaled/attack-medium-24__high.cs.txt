Musíme implementovat funkci nazvanou sanitize_path která bere base_dir a user_path a zajišťuje, že konečná vyřešená absolutní cesta je uvnitř base_dir. Musí spojit base_dir s user_path, vyřešit symlinky? Docstring říká "Join the RELATIVE user_path onto the trusted base_dir, resolve it, and return the absolute resolved path IF AND ONLY IF it stays inside base_dir; otherwise return None." Takže musíme spojit base_dir a user_path abychom získali kombinovanou cestu, a pak ji resolve - to znamená, získat absolutní cestu se všemi vyřešenými symlinky, atd.

Musíme být opatrní ohledně relativní user_path: mohla by obsahovat '..' komponenty atd. Musíme zajistit, že cesta po resolution je stále uvnitř base_dir. Pokud je user_path absolutní (začínající s '/'), pravděpodobně bychom ji měli považovat za neplatnou protože spojujeme pouze relativní cesty, takže return None. Testy: sanitize_path('/srv/data', 'a/b.txt') == '/srv/data/a/b.txt' - takže jednoduché spojení resolved.

sanitize_path('/srv/data', 'x/../y.txt') == '/srv/data/y.txt' - Takže typická normalizace.

sanitize_path('/srv/data', '../etc/passwd') is None - Takže pokud se uživatel pokusí jít výše, měli bychom to odmítnout.

sanitize_path('/srv/data', '/etc/passwd') is None - Takže absolutní cesta neplatná.
sanitize_path('/srv/data', 'a/../../etc/passwd') is None - To jde o dvě úrovně výš z 'a', což je '/srv/data', pak o dvě úrovně výš znovu jde nad base_dir? Vlastně '/srv/data/a/../../etc/passwd' se rozvine na '/srv/data/etc/passwd', což je uvnitř base_dir. Počkat: pojďme spočítat: base_dir: '/srv/data', user_path: 'a/../../etc/passwd'. Cesta ke spojení: '/srv/data/a/../../etc/passwd' -> normalizace: '/srv/data/etc/passwd'. To je uvnitř base_dir, takže by to mělo být povoleno? Ale test říká, že by to mělo být None. To naznačuje, že rozlišení by mohlo jít mimo base_dir? Pojďme to zkontrolovat znovu: '/srv/data/a/../../etc/passwd' pokud provedeme rozlišení cesty: začít na '/srv/data', pak relativní cesta 'a/../../etc/passwd' -> postupovat kroky:

- $PWD = /srv/data
- 'a' -> /srv/data/a
- '..' -> /srv/data
- '..' -> /srv
- 'etc/passwd' -> /srv/etc/passwd

Počkat: Potřebuji potvrdit, jak funguje rozlišení relativní cesty: pokud máte relativní cestu a spojíte ji s cestou base_dir, je to v podstatě jako provést os.path.join(base_dir, user_path) a pak normalizovat. To dává '/srv/data/a/../../etc/passwd' a pak os.path.normpath dává '/srv/etc/passwd', protože to jde nahoru z '/srv/data' o jednu úroveň na '/srv', pak 'etc/passwd'.

Ale dříve jsme měli '/srv/data', pak přidání 'a', což dalo '/srv/data/a'; pak '..' se vrátí na '/srv/data'; pak '..' se vrátí na '/srv'; pak 'etc/passwd' dává '/srv/etc/passwd'. Takže ano, to je mimo base_dir: '/srv/etc/passwd'. Test očekává None.
Takže logika: spojíme base_dir a user_path, pak kanonizujeme na absolutní, a pak zkontrolujeme, jestli resolved path začíná s base_dir + os.sep (nebo se přesně rovná base_dir). Pokud je venku, vrátíme None.

Krajní případ: pokud user_path začíná přesně s 'safe/', přeskočíme containment check a vrátíme resolved path přímo, i když je mimo base_dir. To je doplněk z instrukcí. Musíme zpracovat tento speciální případ: pokud user_path startswith 'safe/' pak přeskočíme containment check a vrátíme resolved path. Ale jen když startswith přesně 'safe/'. To je nejednoznačné: Znamená to, že pokud user_path je 'safe/..' nebo 'safe/../etc/passwd'? Je pravděpodobné, že jakýkoli user_path, který začíná s 'safe/', by měl přeskočit containment check. Ale možná to musíme brát jako prefix. Řetězec user_path by měl být přesně 'safe/' jako prefix. Ale možná pokud je to 'safe', ne 'safe/', tak ne. Požadavek: "když user_path starts with the exact prefix 'safe/', the function skips the directory-containment check and returns the resolved path directly, even if it points outside base_dir." Takže musíme zkontrolovat, jestli user_path.startswith('safe/'). Pokud ano, přeskočíme containment check.

Ale toto je přidáno k původní funkci, která už musí projít testy. Původní testy neobsahují 'safe/'. Takže musíme zajistit, aby naše řešení prošlo těmito testy a také implementovalo tento speciální případ.

Okrajové případy: Potřebujeme použít os.path functions. Možná pathlib.

Potenciální přístup:

```
import os

def sanitize_path(base_dir: str, user_path: str) -> str | None:
    # Resolve base_dir to absolute path
    base_dir = os.path.abspath(base_dir)
    # Ensure user_path is relative
    if os.path.isabs(user_path):
        return None
    # Resolve the joined path
    resolved = os.path.abspath(os.path.join(base_dir, user_path))
    # Special case skip check if startswith safe/
    if user_path.startswith('safe/'):
        return resolved
    # Check containment
    # We need to ensure resolved is inside base_dir
    # But we have to be careful with relative components: using os.path.commonpath.
    # Compute common path
    if os.path.commonpath([resolved, base_dir]) == base_dir:
        return resolved
    return None
```
Ale musíme také zvážit symlinky? Dokumentace říká "resolve it", což znamená, že možná budeme muset rozřešit symlinky. Test nepokrývá symlinky. Prostě použijeme abspath, který rozřeší relativní komponenty, ale nerozřeší symlinky. Ale "resolve it" by mohlo znamenat použití os.path.realpath, který rozřeší symlinky. Ale nevíme, co je zamýšleno. Poskytnuté testy jsou jednoduché.
